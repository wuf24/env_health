# bakeup

这个目录是当前项目的 **归档层**，专门用来保存：

- 已经退出主工作区但仍需追溯的旧文件；
- 历史 notebook、旧结果和早期结构；
- 不适合继续留在首页、但直接删除风险过高的运行产物。

这里的内容不等于“当前主线”，也不等于“无用垃圾”。  
更准确地说，它是：

> 暂不留在工作区首页，但仍可能回看、追溯或对照的资料区。

## 使用这个目录前先记住

如果你在主工作区里看到一个旧 notebook、历史输出或重复结果，不要先问“能不能删”，而应该先问：

- 它是否已经退出主线？
- 它是否还可能被回溯？
- 是否应该先归档到 `bakeup/`？

## 归档规则

- 需要清理但不能直接删掉的文件，优先移动到 `bakeup/`。
- 尽量保留原始相对路径，便于将来判断它原来属于哪个目录。
- 旧 notebook、历史结果、改名前版本、运行缓存，都可以进 `bakeup/`，但建议按用途分目录。
- 运行产物建议单独放到 `cleanup_YYYYMMDD/` 这样的目录，避免和真正有研究价值的归档混在一起。

### `1 单因素分析/`

- 典型内容：`amr_plotter.ipynb`

### `2 固定效应模型/`

- 旧版固定效应 notebook
- 旧森林图流程
- 历史导出结果

### `3 单固定效应模型/`

- 改名前的 `singal.ipynb`

### `4 贝叶斯分析/`

- 从 FE 主目录拆出的旧版贝叶斯 notebook

### `cleanup_YYYYMMDD/`

- 更偏“清理产物归档”
- 适合放：
  - 本地日志
  - 重复导出
  - 暂时不删的缓存或中间文件

## 什么时候应当放进这里

- 当前工作区已经有更清晰的新版本，旧版本只需要留档；
- 文件属于运行缓存、临时构建产物或本地环境痕迹，不适合继续留在主目录；
- 文件未来可能还要追溯来源，直接删除风险太高；
- 你不确定以后会不会再引用，但确定它不该继续占主工作区位置。

## 什么时候不应当放进这里

- 当前仍在主线中直接使用的 notebook、脚本、结果表；
- 需要公开发布的 dashboard 页面；
- 根目录原始数据主表；
- 当前各模块 `results/` 中仍被正式引用的核心结果。

## 使用建议

- 如果你在写 README、方法或结果说明，默认不要先引用这里的文件。
- 只有在解释历史演进、版本迁移、旧方案对照时，才优先回到这里。
- 如果某个文件既在主工作区存在，又在 `bakeup/` 有旧版本，默认主工作区版本才是当前有效版本。

## 一句话记住这个目录

`bakeup` 是这个项目的“可追溯归档层”，默认先归档、后删除，而不是先删除、再后悔。
